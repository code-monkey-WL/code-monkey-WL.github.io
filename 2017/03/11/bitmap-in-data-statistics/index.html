bitmap 在数据统计中的应用

Posted on 2017-03-11 In data

这段时间一直在负责公司内部的数据报表业务，主要就是汇总统计用户的日活、新增、一些页面的点击、UV 等，做成漂亮的报表，供公司的运营、leader 查看，为他们的决策提供数据支持。这些统计其实是很基础的，但是当数据量增大到一定的程度，量变引起质变，就产生了一些问题。

问题

按照一个页面的统计来说，随着用户的增长，每天都可能产生亿级的数据，如果想要统计这个 page 的 天级别的 UV，需要在上亿的数据中去重，这个计算量是很大的，目前的做法是由数据组的同事每天通过 hadoop 集群将前一天的数据处理完毕，插入到 MySQL 数据库，每天就是一条记录，然后我来读取处理成报表需要的数据。与之对应的也有周级别、月级别的报表。

那么想要查询任意时间段的报表怎么办呢？数据组的同事给了一个接口，它实时使用大数据集群计算相应的数据，但是对于譬如三个月的数据查询，就有 100 亿的数据进行 map-reduce ，通常需要几分钟的时间才能得到结果，这种体验着实有点差。

bitmap

在业余时间和同事聊天的得到了启发，bitmap 可以用来进行这种情况的数据统计，效率非常之高，而且非常节省空间。bitmap 是把二进制的 bit 映射到相应的 value 的一种数据结构，因为采用了二进制存储，因此占用空间很小，为了验证它的效率，我写了个 demo 验证一下。

测试

分别进行了多次测试，月级别的查询在毫秒级，一年的数据在 5s 以内就可以查出，考虑到我的电脑只是一个 I5 的低压 CPU，在服务器上运行速度应该更快。

总结

bitmap 在大数据统计的效率确实很高，不但可以方便的统计日活，用户的新增、留存率统计也很方便。但是也有缺点，当数据量足够大的时候，从 MySQL 查出数据在 Server 进行运算的方式并不太好，以 demo 为例，一年大概 365 * 12 = 4G 的数据，如果数据再大点，很容易就导致内存溢出，而且这么大的数据量从 MySQL 传输到 Server 端也是相当耗时的。我觉得有两种方式可以解决：

  • hadoop 集群中存储 bitmap 结构的数据，在集群中进行计算，跳过 MySQL 中间层。
  • 扩展MySQL，使用 c/c++ 编写自定义聚合函数，在 MySQL Server 上进行计算。

后记

这个使用 bitmap 来提搞数据查询效率的想法并没落地，主要是整个实现需要的工作量并不小，而且运营同事对目前的查询效率还可以忍受，因此这个提议目前被搁置了，还是等以后看看能不能在生成环境中验证吧。
